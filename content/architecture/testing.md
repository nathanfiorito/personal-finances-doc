# Testing Strategy

The backend uses three complementary levels of testing. All tests live under `app/src/test/`.

## Test Pyramid

| Level | Tool | Docker needed? | What it covers |
|---|---|---|---|
| Unit | JUnit 5 + AssertJ + Mockito | No | Use cases, domain logic, edge cases |
| Integration | JUnit 5 + Testcontainers | Yes | Repository adapters against a real PostgreSQL container |
| Architecture | ArchUnit | No | Hexagonal layer boundaries |

Run all three with:

```bash
cd app && mvn verify
```

Run only unit tests (no Docker):

```bash
cd app && mvn test
```

Coverage reports are generated by JaCoCo (`target/site/jacoco/index.html`).

## Unit Tests

Located under `src/test/java/.../application/` and `src/test/java/.../domain/`.

### Transaction use cases

| Test class | Covers |
|---|---|
| `CreateTransactionUseCaseTest` | Happy path, category not found |
| `GetTransactionUseCaseTest` | Found, not found (→ `TransactionNotFoundException`) |
| `ListTransactionsUseCaseTest` | Pagination defaults, boundary values |
| `UpdateTransactionUseCaseTest` | Partial updates, not found |
| `DeleteTransactionUseCaseTest` | Found, not found |
| `GetSummaryUseCaseTest` | Date range filtering, type filter |
| `GetMonthlyUseCaseTest` | Year grouping, category breakdown |
| `ExportCsvUseCaseTest` | CSV header, BOM present, row formatting |

### Category use cases

| Test class | Covers |
|---|---|
| `CreateCategoryUseCaseTest` | Happy path |
| `UpdateCategoryUseCaseTest` | Happy path, not found |
| `DeactivateCategoryUseCaseTest` | Happy path, not found |
| `ListCategoriesUseCaseTest` | Pagination, activeOnly flag |

### Telegram use cases

| Test class | Covers |
|---|---|
| `ProcessMessageUseCaseTest` | Text/image/pdf entry types, LLM extraction, categorisation, pending state set, confirmation message sent |
| `ConfirmTransactionUseCaseTest` | No duplicate → save, duplicate detected → warning, `skipDuplicateCheck=true` → save without check, expired state |
| `CancelTransactionUseCaseTest` | State cleared, message edited |
| `ChangeCategoryUseCaseTest` | Category updated in pending state, message edited, missing/expired state returns false |

### Domain records

| Test class | Covers |
|---|---|
| `ExtractedTransactionTest` | Amount validation (negative, zero, over limit), CNPJ formatting |

All use case tests use Mockito mocks for ports — no Spring context is loaded.

## Integration Tests

Located under `src/test/java/.../infrastructure/`.

Tests that require a real PostgreSQL instance use **Testcontainers** (`@Testcontainers` + `@Container`). A `postgres:16` container is started automatically and a clean schema is applied via Flyway before each test class.

### What is integration-tested

- `TransactionRepositoryAdapter` — all `TransactionRepository` methods against real SQL
- `CategoryRepositoryAdapter` — all `CategoryRepository` methods against real SQL

**Not** integration-tested (tested at unit level or manually):
- `OpenRouterLlmAdapter` — hitting a real LLM is too slow/expensive for CI
- `TelegramNotifierAdapter` — requires live Telegram credentials

## Architecture Tests (ArchUnit)

`HexagonalArchitectureTest` enforces the hexagonal dependency rules at build time:

```
Domain    ──  no dependency on application, infrastructure, or interfaces
Application ──  no dependency on infrastructure or interfaces
Infrastructure ── implements domain ports (secondary adapters)
Interfaces ── depends only on application (use cases)
```

Concrete rules enforced:

```java
// Domain must not use Spring annotations
noClasses().that().resideInPackage("..domain..")
    .should().dependOnClassesThat().resideInPackage("org.springframework..")

// Application must not reach into infrastructure
noClasses().that().resideInPackage("..application..")
    .should().dependOnClassesThat().resideInPackage("..infrastructure..")

// Domain must not reach into application or infrastructure
noClasses().that().resideInPackage("..domain..")
    .should().dependOnClassesThat().resideInPackage("..application..")
    .orShould().dependOnClassesThat().resideInPackage("..infrastructure..")
```

If any of these rules are violated, `mvn test` fails with a clear violation report.

## Security Tests

Spring Security configuration is tested via `@SpringBootTest` + `MockMvc` (or `WebMvcTest`) slices. These verify:

- `POST /api/auth/login` with valid credentials → 200 + JWT
- `POST /api/auth/login` with wrong password → 401
- `GET /api/v1/transactions` without JWT → 401
- `GET /api/v1/transactions` with valid JWT → 200
- `POST /webhook` without secret token → 403
- `POST /webhook` with valid secret → 200

## Test Dependencies (pom.xml)

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>  <!-- JUnit 5, AssertJ, Mockito -->
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>postgresql</artifactId>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>com.tngtech.archunit</groupId>
    <artifactId>archunit-junit5</artifactId>
    <version>1.4.1</version>
    <scope>test</scope>
</dependency>
```

Coverage reporting:

```xml
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.13</version>
</plugin>
```
